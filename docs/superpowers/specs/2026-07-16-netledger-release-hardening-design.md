# NetLedger 全面优化设计说明

## 1. 目标与边界

本轮目标是在不重写现有 React + FastAPI 主体的前提下，把 NetLedger 从“可演示的毕业设计”提升为“可安全部署、可维护、可验证的轻量 IPAM”。优化覆盖产品表达、UI/响应式、认证安全、输入校验、数据迁移、测试、容器部署和运维文档。

本轮不引入在线扫描、OIDC、Redis 集群、IPv6 地址池或计费等新业务。这些能力属于后续独立子项目，不应混入发布加固。

## 2. 业务分析

### 2.1 用户与痛点

- 系统管理员：需要安全初始化账号、管理用户、审计变更，并能在新环境可靠部署。
- 网络管理员：需要快速查看地址利用率、维护站点/子网/设备、处理冲突。
- 部门申请人：需要在授权范围内分配地址，不能越权查看或修改其他部门资源。
- 只读访客：需要稳定查看台账和统计，不应看到管理入口。
- 运维/接手开发者：需要明确的环境变量、迁移、备份、测试与回滚步骤。

当前主要痛点是：界面仍暴露“毕业设计”和默认账号；移动端菜单缺少退出入口与可访问名称；表单标签没有与控件关联；生产环境仍会自动写入演示账号；密码修改后旧 JWT 仍有效；限流把成功登录也计入失败次数；Alembic 与当前模型不一致；容器未执行正式迁移且后端以 root 运行；前端没有自动化测试脚本。

### 2.2 商业价值

- 降低默认口令、令牌复用、错误代理信任造成的安全风险。
- 提升移动运维、键盘操作和辅助技术用户的可用性。
- 让演示环境与生产环境明确分离，提升产品可信度。
- 让数据库升级、容器部署、备份和回滚可重复执行，降低交接成本。

### 2.3 核心流程

1. 首次启动：开发环境可显式加载演示数据；生产环境只允许使用强密码引导管理员，缺少引导配置时快速失败。
2. 登录：校验限流状态 → 验证凭据 → 成功后清理失败记录 → 生成带认证版本的短期 JWT。
3. 已认证请求：校验签名、有效期、用户状态、角色和认证版本；密码修改/重置后旧令牌失效。
4. 日常运维：工作台查看风险 → 站点/子网/地址/设备处理 → 写入审计日志 → 冲突闭环。
5. 发布：安装运行时依赖 → Alembic 升级 → 非 root 启动 API → Nginx 提供 SPA、缓存和安全头 → 健康检查。

## 3. 功能拆解与数据流

### 3.1 产品与 UI

- 品牌层
  - 输入：产品名、环境模式。
  - 输出：NetLedger 品牌、环境徽标、部署中立文案。
  - 依赖：Vite 环境变量。
- 应用壳层
  - 输入：当前用户、角色、开放冲突数。
  - 输出：桌面侧栏、移动抽屉、用户会话与退出操作。
  - 数据流：AuthContext → AppShell → 导航/会话操作。
- 表单与反馈层
  - 输入：字段定义、验证结果、异步状态。
  - 输出：关联 label/id、错误提示、加载/成功状态。
  - 依赖：React `useId`、统一 UI 组件。
- 响应式层
  - 输入：375、768、1440 三档视口。
  - 输出：无横向溢出、触控目标不小于 44px、移动端可完成退出与导航。

### 3.2 认证与安全

- 登录限流
  - 输入：可信客户端 IP、失败事件。
  - 输出：剩余尝试或 HTTP 429。
  - 数据流：Request → 代理策略 → 失败桶；成功登录清桶。
- 密码策略
  - 输入：新密码。
  - 输出：通过或结构化 422；要求至少 12 字符并包含字母、数字和特殊字符，且不超过 bcrypt 72 字节。
- 令牌版本
  - 输入：用户 `auth_version` 与 JWT `ver`。
  - 输出：有效用户或 401。
  - 数据流：登录签发版本 → 请求比对版本 → 改密/重置递增版本。
- 安全响应头
  - 输入：所有 HTTP 响应。
  - 输出：CSP、nosniff、frame deny、referrer policy、permissions policy、request id。

### 3.3 数据与部署

- 数据库迁移
  - 输入：现有 0001 schema。
  - 输出：补齐 devices、device_id、auth_version 和唯一 MAC 索引的 0002 迁移。
  - 依赖：Alembic、SQLAlchemy。
- 初始化
  - 输入：APP_ENV、SEED_DEMO_DATA、BOOTSTRAP_ADMIN_*。
  - 输出：开发演示数据或生产管理员；禁止生产默认口令。
- 容器启动
  - 输入：环境变量、数据库连接。
  - 输出：迁移成功后的非 root API 与带缓存/安全头的 SPA。

## 4. 方案对比

### 方案 A：仅视觉优化

改动小、展示快，但无法解决默认账号、旧令牌、迁移和部署缺口，不满足“整体优化”。不采用。

### 方案 B：现有架构发布级加固（采用）

保留 React/Vite、FastAPI/SQLAlchemy 和现有 REST API，增加品牌系统、可访问性、认证版本、生产引导、迁移与容器加固。风险可控，现有 32 项测试可作为回归基线。

### 方案 C：改用 Next.js/PostgreSQL/Redis 全量重构

长期扩展性更强，但会重写认证、路由、数据层和部署，超出本轮发布加固，且没有业务证据证明重构收益大于迁移风险。不采用。

## 5. 技术方案与架构

### 5.1 技术栈

- 前端：React 19、TypeScript、Vite 8、Tailwind CSS 4。沿用现有依赖，避免引入网络安装风险。
- 后端：Python 3.11、FastAPI、SQLAlchemy 2、Pydantic 2、Alembic。
- 数据库：SQLite 用于开发/小型单机；PostgreSQL 用于生产，补充 psycopg 驱动和正式迁移流程。
- 部署：Docker Compose、Nginx；后端非 root，启动前执行 Alembic。
- 测试：pytest + Node 内置 test runner + TypeScript 类型检查 + 浏览器交互验收。

### 5.2 请求架构

```text
Browser
  -> Nginx (SPA / cache / CSP / request limits)
  -> FastAPI (request id / security headers / CORS)
  -> Router + RBAC + Pydantic validation
  -> Domain services
  -> SQLAlchemy
  -> SQLite or PostgreSQL
```

日志由 API 访问日志和业务 AllocationLog 构成；健康检查验证数据库；生产监控以 `/health`、容器健康状态和结构化日志为基础。

## 6. UI 设计系统

- 定位：克制、可信、带网络运维感，不使用通用紫色 AI 风格。
- 配色：深墨蓝导航、翡翠绿主操作、琥珀色风险提示、暖灰纸面背景。
- 字体：中文标题使用本地衬线显示栈，正文使用清晰的系统无衬线栈；数字与 CIDR 使用等宽字体。
- 形态：保持非对称工作台和深浅分区；增加细网格/径向光晕层次，避免纯白平铺。
- 交互：统一 focus-visible、hover、disabled、loading、error、success 状态；遵守 reduced-motion。
- PC：固定 232px 侧栏，内容区最大宽度控制，主要统计为不对称网格。
- 平板：隐藏侧栏，使用抽屉导航，表格允许局部横向滚动。
- 手机：单列卡片，顶部操作压缩，抽屉包含用户身份与退出，44px 触控目标，无整页横向滚动。

## 7. API 与数据库变化

### 7.1 API 行为

- `POST /api/v1/auth/login`：只记录失败尝试；成功清除当前客户端失败记录。
- `POST /api/v1/auth/change-password`：强密码校验；成功递增认证版本，旧 JWT 在下一次请求返回 401。
- 管理员重置用户密码：同样递增认证版本。
- 所有响应增加安全头；错误体继续使用现有 `detail` 结构，保持前端兼容。

### 7.2 数据变更

- `users.auth_version INTEGER NOT NULL DEFAULT 0`，索引不需要。
- `devices.mac` 使用唯一索引，允许多个 NULL，消除并发创建重复 MAC 的竞态。
- 0002 迁移补齐现有 0001 未包含的 devices 与 ip_addresses.device_id。

## 8. 异常与安全设计

- SQL 注入：继续只使用 SQLAlchemy 参数化表达式；不拼接用户 SQL。
- XSS：React 默认转义；CSP 限制脚本源；不使用 `dangerouslySetInnerHTML`。
- CSRF：当前 API 使用 Authorization Bearer，不依赖 Cookie，因此无环境自动附带凭据；CORS 使用明确来源。
- Token 伪造：强 SECRET_KEY、固定 HS256、exp、sub、ver 校验；密码变化撤销旧令牌。
- 暴力破解：只统计失败登录；生产多实例部署仍建议 Redis 统一限流。
- 权限绕过：后端 RBAC 是唯一授权边界，前端隐藏按钮只用于体验。
- 文件上传：保留 CSV 类型/内容校验，并由 Nginx 限制请求体。
- 敏感信息：生产不渲染演示账号，不自动创建默认口令，不在日志输出密码或令牌。
- 代理欺骗：默认不信任 `X-Forwarded-For`；仅在显式 TRUST_PROXY_HEADERS 时启用。

## 9. 性能设计

- 修复站点列表逐站点 count 的 N+1 查询，改为分组聚合。
- 静态资源使用内容哈希长期缓存，HTML 禁止强缓存并启用 gzip。
- 地址池继续限制自动展开规模；分页上限保留，避免大响应。
- 不在本轮引入 Redis；多实例限流和看板缓存列为后续扩展点。

## 10. 测试与验收

- 单元/接口：弱密码拒绝、成功登录不消耗失败额度、失败登录限流、改密撤销旧令牌、生产初始化安全、响应安全头。
- 前端：日期/百分比等纯函数 Node 单测；TypeScript build；oxlint 零告警。
- 集成：32 项原有回归测试继续通过；新增测试通过；健康检查和登录主流程通过。
- UI：浏览器验证登录、工作台、站点表单、移动抽屉；375/768/1440 无整页横向溢出；交互控件有可访问名称，字段 label 可关联。
- 部署：Docker 配置可解析；若本机 Docker 可用，执行镜像构建和 Compose 健康检查。
